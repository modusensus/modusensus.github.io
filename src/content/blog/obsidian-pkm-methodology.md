## 为什么需要一套自己的架构

我用 Obsidian 两年，前一年半都在"堆笔记"——上课记、读书记、刷论文记，记完就放在那里。等到考研复习时想找"新公共管理的发展脉络"，打开搜索，出来 47 条结果。没有层级，没有关联，没有入口。

问题不在 Obsidian，在我没有给它一个**结构**。

后来我花了三个晚上重新设计文件夹结构。核心原则很简单：**不要让分类成为记录的阻力，让整理成为 AI 和流程的事**。

---

## 七层架构

```
MyBrain/
├── 00-Inbox       ← 一切从这里开始
├── 01-AI          ← AI 工具、Prompt 库、工作流
├── 02-Projects    ← 有截止日的项目
├── 03-Study       ← 考研主战场
├── 04-Life        ← 生活事务
├── 05-Archive     ← 已完成、不再改
├── 99-Ideas       ← 种子库
└── README         ← 本文件的说明
```

---

### 00-Inbox · 收件箱

**零摩擦入口。不需要分类，不需要标题，不需要标签。**

任何想法、截图、链接、课堂随手记，全部先扔进 Inbox。分类是后置的——每天晚上或隔天，花 5 分钟把 Inbox 的内容搬到对应文件夹。

这个设计的灵感来自 GTD 的收件箱概念，但更重要的是一个心理效果：**你不会因为"不知道放哪"而干脆不记**。

规则只有一条：Inbox 不能超过 20 条。超过了就必须清。

---

### 01-AI · AI 工具与工作流

AI 不是"一个工具"，它是一个**独立的认知层**。所以我把它单独成区，而不是放在 02-Projects 下面。

这个文件夹里放的是：

- **Prompt 库**：按场景分类的提示词模板（论文拆解、笔记整理、复习提纲生成）
- **工作流文档**：n8n 自动化配置、DeepSeek API 调用方式、Claude Code 常用命令
- **工具对比**：不同 LLM 提供商的 API 价格、延迟、中文能力实测记录
- **CLI 工具笔记**：下面会详细讲

---

### 02-Projects · 项目

有明确产出和截止日的东西。用 **GAP 追踪法**：每个项目开头写三行——

```
🎯 Goal:   我要做成什么
📍 Actual: 现在做到哪了
📋 Plan:   下一步做什么
```

当前活跃的项目：CaseForge、MindOS、博客网站、考研复习计划。

项目完成后整体迁移到 05-Archive。

---

### 03-Study · 学习

考研的主战场。按学科分子文件夹：

```
03-Study/
├── 631-公共管理学/
│   ├── 核心概念/
│   ├── 理论流派/
│   └── 真题解析/
├── 864-公共政策学/
├── 政治/
└── 英语/
```

每门课的笔记模板统一：**概念定义 → 理论脉络 → 典型案例 → 真题链接**。这个模板固定在 Obsidian 的 Templates 插件里，新建笔记一键插入。

---

### 04-Life · 生活

旅行计划、健康记录、财务追踪、读书清单。和知识体系无关但需要管理的事务。

---

### 05-Archive · 归档

**已完成、不再活跃的项目和学习模块。** 不是垃圾桶——归档的内容保留完整结构和标签，只是从活跃区移出，减少日常浏览的噪音。

比如重修考试结束后，那部分复习笔记就整体迁到 05-Archive/重修考试/。

---

### 99-Ideas · 种子库

这是整个知识库里最特别的一个文件夹。

和 00-Inbox 不同：Inbox 是"待处理的信息碎片"，99-Ideas 是"我主动产生的想法"——想做的项目、想写的文章选题、想研究的问题。

想法在这里孵化。当一个想法积累到足够的笔记和链接（通常 3-5 条关联笔记），就升级为一个 02-Projects 里的正式项目，或者扩展为一篇博客文章。

**99 号的意义**：放在最下面，不是为了隐藏——是因为它离"完成"最远，离"可能性"最近。

---

## CLI 工具集成：让 AI 进入知识库

我在这套系统里接入了两个命令行 AI 工具：**Claude Code** 和 **Codex**。它们的角色不同。

### Claude Code 的角色

Claude Code 直接读取 Obsidian 的 Markdown 文件，我主要用它做三件事：

**1. 整理笔记结构**

在 Inbox 里堆了十几条零散笔记后，对 Claude Code 说：

```
把 00-Inbox 里的笔记按主题分类，合并重复内容，
生成结构化的笔记放到 03-Study 对应位置。
```

它直接读写文件，一步到位。比手动拖拽快得多。

**2. 生成复习提纲**

考研阶段最常用的命令：

```
读取 03-Study/631-公共管理学/ 下所有笔记，
生成一份包含核心概念、对比表格、填空题的复习提纲。
```

输出直接写成一个新的 Markdown 文件，我第二天早上对着复习。

**3. 交叉链接**

```
扫描 03-Study 下的所有笔记，
找出可以互相关联的概念对，
自动添加 [[双向链接]]。
```

这个功能让孤立的知识碎片变成了网络。比如"新公共管理"和"企业家政府"这两篇笔记之间，Claude Code 能自动识别关联并加上链接。

### Codex 的角色

Codex 和 Claude Code 的定位不同：Claude Code 做**知识处理**，Codex 做**自动化脚本**。

比如我想把 Inbox 清空流程自动化：每次运行脚本，扫描 00-Inbox，根据内容自动建议移动到哪个文件夹。这个脚本就是让 Codex 帮忙写的。

两者的协作模式：

```
Claude Code  →  理解内容、整理知识、生成提纲
Codex        →  写脚本、自动化重复操作
我           →  做判断、选方向、写创造性的内容
```

## 这套系统的设计原则

回头总结，这套架构背后有五个原则：

**1. 零摩擦捕获（00-Inbox + 99-Ideas）**

想法来的时候不要挡它。Inbox 接住一切外部信息，Ideas 接住一切内部灵感。分类是之后的事。

**2. 分类后置**

不要让用户（你自己）在记录时做分类决策。让流程和 AI 在整理时做。这解决了我之前"不知道放哪就不记了"的问题。

**3. 活跃与归档分离（05-Archive）**

活跃项目和已完成项目放在一起会制造噪音。Archive 的存在让日常浏览只看到"现在在做的"，而不是"曾经做过的"。

**4. AI 是第一读者**

这套结构的文件夹命名和模板设计，不只是给我自己看的，也是给 Claude Code 读的。清晰的层级让 AI 能更准确地理解上下文，生成更相关的整理建议。

**5. 简单到可以手绘**

如果有人问你这套系统怎么用，你应该能在一张餐巾纸上画出来。七个文件夹，一个 README。不需要插件、不需要复杂配置，新电脑上 5 分钟搭好。

---

## 不是什么

最后说三个"不是"：

- **不是 PARA 的复刻**：PARA 的分类逻辑启发了我，但我加了 99-Ideas 和 01-AI，因为这两个需求在传统知识管理框架里没有对应。
- **不是 Notion 替代品**：Obsidian 做的是本地 Markdown 文件，Notion 做的是云数据库。我选 Obsidian 是因为文件在本地、CLI 工具能直接读写。
- **不是"最好"的系统**：这是为**跨学科学习者 + AI 重度用户**这个特定画像设计的。如果你的工作流不同，改它。
